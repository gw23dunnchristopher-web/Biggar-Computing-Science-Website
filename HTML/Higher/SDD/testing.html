Comprehensive Testing Plans

A comprehensive final test plan should ensure that the functional requirements are met. A test plan can be created after the analysis stage. Testing should include normal, exceptional and extreme data. It should also ensure that every line of code has been tested.

Debugging Techniques

Debugging is the process of finding and fixing errors in a program. It is an important skill for a programmer to have, and is a key part of the testing process. At Higher level, you have to be aware of the following debugging techniques:

Dry Run▶

A dry run is a paper-based analysis of code. A programmer...

  1. Looks at the program on the screen or at a print out of the code.
  2. Uses a pen and paper and executes the program line by line in their head.
  3. Notes down the changes to values of each variable in a trace table.
  4. Checks that the variables are used and updated as expected.

This helps to find logic and execution errors and can be carried out during design, implementation, testing or maintenance.

Trace Tables/Tools▶

Each programming environment has a feature that shows a table of values revealing the contents of several variables as each line stepped through. Trace tables can also be created on paper as part of a dry run, showing how the variables change as the user moves through the program.

Breakpoints▶

Pre-defined points can be set in a program that allows the execution of code to be stopped so the programmer can examine the values stored in variables at that time and compare them with the expected values.

In Thonny, Breakpoints are represented by a red dot. One is shown here at line 6.

Watchpoints▶

A special breakpoint that stops your program when the value of a variable changes or a pre-set condition is met. The stored values can be compared with what the programmer expected. Not all programming environments use watchpoints, in this case it is common practice to temporarily use print statements that display the changing value as the program runs.

On Debugging Large Programs

When debugging large programs...

  • A trace table may be used to find out which value is not as it should be.
  • The programmer can then set breakpoints to stop/pause program at defined points.
  • The values of the variables are compared with the expected value in the trace table to see if they match.
