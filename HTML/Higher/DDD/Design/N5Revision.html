N5 Database Design Recap

Database Terminology

The following terms were covered at N5 level, and are still used at Higher level.

TermMeaning
Entity

An entity is a person, place, thing, event or concept of interest to the business or organisation about which data is to be stored. For example, in a school, possible entities might be Student, Teacher, Class and Subject.

Attribute

An attribute is a property or characteristic of an entity. For example, attributes of a Student entity might be StudentID, First_Name, Surnname, and dateOfBirth.

Primary Key

A primary key is an attribute or combination of attributes that uniquely identifies a record in an entity. For example, StudentID is a primary key for the Student entity. A primary key is signified by underlining in an entity-relationship diagram.

Foreign Key

An attribute in one table that uniquely identifies a row of another table. A foreign key is signified by an asterisk in an entity-relationship diagram.

Relationship

A relationship is an association between two or more entities. For example, Students learn from Teachers and Teachers educate Students.

Cardinality

The cardinality of a relationship defines the number of participants in the relationship. It states the number of entity occurences in one entity that are associated with a single occurance of the related entity.

Data Dictionaries

Data Dictionaries were covered at the N5 level, and they are still used in Higher. While the information shown in a data dictionary is the same, you will now be expected to be able to produce a data dictionary showing at least 3 entities.

A data dictionary should contain:

  • The name of each entity.
  • The name of each attribute within each entity.
  • The type of data that will be held by each attribute.
  • The size of each attribute (for text attributes).
  • Indication of attributes that will be used as Primary or Foreign keys.
  • Validation rules that will be applied to any attributes.

Validation

Here is a small review of the types of validation found in N5.

  • Presence Check - Checks to make sure the user has entered data in the field before it allows them to create a record.
  • Range Check - Checks to make sure that the number entered is within a certain range.
  • Length Check - Checks to make sure that the user has entered a certain number of characters.
  • Restricted Choice - Gives the user a list of options to choose from.

Entity Relationship (ER) Diagram

An Entity Relationship (ER) Diagram is a graphical representation of the relationships between tables that make up a database.

  • Entities - The entities within the ER diagram are the names of the tables that will make up the database.
  • Attributes - The attributes are the names of the fields that will be in the tables.
  • Primary Key - An attribute that uniquely identifies each record in an entity.
  • Foreign Key - An attribute that is a Primary Key in a different entity, linking the two entities together.

Cardinality

Cardinality is the number of participants in a relationship. There are three types of cardinality:

  • One-to-one
  • One-to-many
  • Many-to-many

The Er Diagram above shows examples of One-to-Many relationships, because one teacher can teach many classes and one class can contain many students. We studied One-to-Many relationships in N5, but at Higher level we must also be aware of Many-to-Many relationships.

One-to-One

In a one-to-one relationship, each entity occurrence in an entity is associated with one, and only one, entity occurrence within a related entity. For example, a School is managed by one, and only one, Headteacher, with a Headteacher managing one, and only one, School.

One-to-Many

In a one-to-many relationship, each entity occurrence in an entity can be associated with one or more entity occurrences in a related entity. For example, a School employs many Teachers and each of those Teachers is employed by one School.

Many-to-Many

In a many-to-many relationship, several entity occurrences in an entity can be associated with multiple entity occurrences in a related entity. For example, many Students study several different Subjects and each of those Subjects is studied by many Students. Direct many-to-many relationships between two entities cannot be implemented by a relational database system. To overcome this problem, many-to-many relationships can be resolved to form two one-to-many relationships.
