Testing

Once a program has been completed, it must be tested in order to find out if there are any bugs that aren't obvious at first glance. If there are, then it may be necessary to return to the implementation stage in order to fix them. Testing is done using different input values to make sure the program works, and there are three types of values that are used:

Normal TestingThese are typical values that you would expect the program to handle. For example, in a program that expects an input of a number between 1 and 10, this would be any number between 2 and 9.
Extreme TestingThese are values which are the boundaries of the acceptable input range. For example, in a program that expects an input of a number between 1 and 10, the extreme values would be 1 and 10. If the program expectes a string, the extreme values may be an empty string or a very long string (depending on the context of the program).
Exceptional TestingThese are values that the program would not be expected to accept. An invalid or unexpected input. For example, in a program that expects you to enter a number between 1 and 10, an exceptional value would be a string, like "one", or a number higher than 10 or less than 1.

Test Data Table

A test data table is used when planning how to test a program. At this level you would typically see a table with six entries - two normal values, two extreme values and two exceptional values. However, at Higher level and above there may be more so that the program is more thoroughly tested.

Here is an example of a Test Data Table for a program that takes in two numbers between 1 and 5, then adds them together.

Type of Test Data Test Data Expected Output Actual Output Comment
Normal 2,3 5 5 Works as intended
Normal 4,2 6 6 Works as intended
Extreme 5,5 10 10 Works as intended
Extreme 5,1 6 6 Works as intended
Exceptional -2,6 Error Message Error Message Works as intended
Exceptional "one", "three" Error Message Error Message Works as intended

Note that in the Exceptional cases, the program is still working as intended by producing an error message. This means that it is catching the errors rather than just crashing.

If the program produces an output that you do not expect at this stage, then that means there is a problem with the code and you would have to go back to the implementation stage and review it.

Errors

An error in programming occurs when something goes wrong that prevents the code from executing as intended. There are three types of errors: Syntax, Logic and Execution (Runtime).

Syntax Errors

These occur when the rules of the programming language are violated, like improper indentation, a missing colon or using incorrect keywords. The code won't even run until they're fixed.

Your programming environment (Thonny, Python IDE, Replit, etc.) has various tools to help you spot errors:

  • When you try to run the program, it will crash and the error may be highlighted.
  • You will receive a comment from the program in the console about the possible location of the error.
  • Your coding environment may automatically indent code on the following line when you press Enter.
  • Variable names may be colour coded so you can see where a variable appears that hasn't been used before (i.e. incorrectly spelled).
  • When there is an error, the programming environment may flag multiple locations where the error may have occured.

Logic Errors

These occur when the program runs, but does not produce the expected output. For example, if you use AND instead of OR in an IF statement, the program would still run, but the output would be incorrect.

It could be that the lines of the program are in the wrong order, or a variable has been initially set to an incorrect value.

Execution (Runtime) Errors

In the case of these errors, the program crashes immediately when it is run when it tries to do something that it cannot possibly do. For example, if the program were asked to divide bby 0, it would try and then crash because that's not possible. If it were asked to peform a function on a string that can only be performed on integers (division for example) then it would try and fail, causing an execution error.

In these cases, both the logic and the syntax are correct, its just that the program cannot do what you are asking it to. The programmer would only discover errors like this when they attempt to run the program.
