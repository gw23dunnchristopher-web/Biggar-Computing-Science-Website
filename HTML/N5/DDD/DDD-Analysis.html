End-User and Functional Requirements

In the Analysis stage of database development, it is important that we identify the End-User Requirements and the Functional Requirements.

End-User Reqirements

  • The end users are the people who are going to be using the database.
  • Their requirements are the tasks they expect to be able to do using the database.

Functional Requirements

  • Processes and activities that the database has to perform.
  • Information that the system has to contain to be able to carry out its functions.
  • The Functional Requirements are usually based on the End-User Requirements.

Example End-User Requirements

A used-car dealership has six showrooms in different locations across Scotland. It wants to create a relational database to store details of cars owned by the company and details of each of their showrooms. The database will allow sales staff to view details of specific cars and to see which showroom the cars are located in.

The developers have asked some of the sales staff what they would like to see in the database. Here are some of their comments:

We can establish the following user requirements from these comments:

  • Sales staff should be able to display details of cars by performing a number of different searches using...
    • Car make and model
    • Car colour
    • Range of car prices
    • Type of fuel used
  • Search results should display...
    • Car make
    • Car model
    • car price
    • Car colour
    • Car fuel
    • Car mileage
    • Branch Address
    • Branch Telephone Number
  • Users should be able to sort the search results in ascending order of mileage and ascending order of price.

Example Functional Requirements

The database will be a relational database with two tables (files), one for the cars and one for the store branches.

Each table will require a field that will be used as a Primary Key. For the cars it will be Car ID, and for the branches it will be Branch ID.

Branch ID will be used in the Car table as a Foreign Key, linking the two tables together.

The database should be able to...

  • filter cars, showing only those within a certain price range.
  • filter cars by make and model.
  • list cars in order of price (with the cheapest first).
  • store the fuel type of each car - deisel, petrol or electric.
  • store the colour of each car.
  • perform a complex sort, listing cars in descending order of mileage and ascending order of price.

Additional fields will be needed for:

  • Car make
  • Car model
  • car price
  • Car colour
  • Car fuel - deisel, petrol or electric.
  • Car mileage
  • Branch Address
  • Branch Telephone Number
