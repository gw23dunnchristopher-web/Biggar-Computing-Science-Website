Testing

How do you test a database?

Once you have implemented your database, it must be tested to make sure that it works as expected and meets the requirements that was set out in the Analysis phase. The following features must be tested:

  • Fields - The data must be checked to ensure that the correct fields are being displayed.
  • Data validation - Any data validation that has been set up must be tested to ensure that it works as expected.
  • SQL Operations - The database must be tested using SQL operations to show that they work as expected and provide the correct output.

Fields

When testing the fields, you must check that the correct fields are being displayed and that the data is being displayed in the correct format. For example, if you have a date field, you must check that the date is being displayed in the correct format (e.g. dd/mm/yyyy). You should also check to make sure that there aren't any unnecessary fields or data that has been duplicated in multiple tables.

Data Validation

Remember that there are four types of data validation:

  • Presence Check - Try to create a record without typing any data into the field with the presence check, it should not allow you to do so.
  • Range Check - Try to enter data outside the lower and upper ranges of a field with a range check, the program should give you an error message.
  • Length Check - It is not enough to just change the field length of a field, since this will only set a maximum number of characters. You should make sure that the user cannot enter any less than the requisite number of characters too using the len() function in the Design View. If they do not enter the right amount of characters, it should display an error message.
  • Restricted Choice - Ensure that the correct choices are being displayed on fields with a restricted choice, and that the user cannot enter any data other that that which appears on the list.

You must test each of these to make sure that they work as expected.

SQL Operations

Even if an SQL Operation has been processed successfully, you must check the results to ensure that it is giving you the correct output.

  • Inappropriate data - Check the output to ensure that the data produced is what you would expect. For example, if you were to perform the SQL Search below on the Theme Parks database you would not expect any rides other than Rollercosters to be displayed.

    SELECT * FROM Ride WHERE Type = 'Rollercoaster';

  • Sorted Correctly - If you have used ORDER BY to apply a sort to your data, you must check to make sure that the data is being displayed in the correct order. One of the most common mistakes in this case is to accidentally add a space as the first character in data. When sorting that field, the database will place any records that contain this space before the others in the table, displaying them in an incorrect order. For example, the table below from the music store database has been sorted using the following SQL statement. Space have been added to some of the titles, so you can see that it appears out of order.

    SELECT * FROM CD ORDER BY Title ASC;

  • Data Range - If you range a query expecting it to return data that is within a range, ensure that the data returned is actually withing that range. For example, if you were to use the SQL query below on the Music Store database, you would expect the following result.

    SELECT * FROM CD WHERE cost >= 7.5 AND cost <= 9,99

    ORDER BY cost ASC;

Checking the Source Data

Once method you could use to help you test the database is to check the source data. This is the data from the full table before it has been sorted or filtered. MS Access provides easy access to the source data just by double clicking on the table name on the left hand side of the screen to open that table. You can then compare the data in the table to the results from any queries you make to make sure that the result of the query is what you would expect.

Temporary Filters

You can also user a temporary filter on the data in the table using the MS Access GUI, as shown in the 'MS Access - Sorting and Searching' page. This will allow you to show only the records containing relavant data before you create a query, and you may be able to see if there are any problems with the data.

Other Tests

You may also want to test the following:

  • Relationships - Check that the relationships between tables are correct and that the correct fields are being used as Primary and Foreign Keys. You can check this by attempt to enter data into the foreign key which has not been entered into the primary key in the corrected table. If you are able to do so, it would suggest that your database does not have referential integrity as every foreign key value has to be a value in the other table's primary key.
  • Primary Keys - Check that the primary key is unique for each record in the table. You can do this by attempting to enter a duplicate primary key value into the table. If you are able to do so, it would suggest that your database does not have entity integrity as every primary key value has to be unique.
  • Data Range - Use a wide range of data to test your data validation.
