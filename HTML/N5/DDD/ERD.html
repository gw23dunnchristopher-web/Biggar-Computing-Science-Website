Entity Relationship Diagrams

What is an Entity Relationship Diagram?

An Entity Relationship (ER) Diagram is a graphical representation of the relationships between tables that make up a database.

  • Entities - The entities within the ER diagram are the names of the tables that will make up the database.
  • Attributes - The attributes are the names of the fields that will be in the tables.
  • Primary Key - An attribute that uniquely identifies each record in an entity.
  • Foreign Key - An attribute that is a Primary Key in a different entity, linking the two entities together.

Relationships

The relationship between two entities show how the tables are linked together. There are three possible descriptions of this relationship, but you will only be using one of these in N5.

The three descriptions are:

  • One to one
  • One to many
  • Many to many

One to One

This is when the relationship between two entities implies that one record in an entity will be connected one record in another entity. For example, if there were a database with one table containing information on people and a second table containing information on passport numbers, that would be a one to one relationship, because each person can only have one passport, and a valid British passport will only be about one person.

One to Many (or Many to One)

This is when the relationship between two entities implies that one record in one table could be linked to many records in another table (or vice versa). For example, if we consider a database containing a table of students a table of library books, one student could borrow many books from the library, but one book could not be borrowed by more than one student at a time.

Many to Many

This is when both entities have many examples of each other. For example, in a database could include a table of teachers and a table of classes. Each teacher could teach many classes, and each class could have many teachers. This is normally considered to be a bad relationship between entities though, and it would normally be better to insert a third entity between the two entities.

At the N5 level, you will only be expected to use the One to Many relationship.

Primary and Foreign Keys

When creating an ER diagram, you will need to identify the Primary Key and any Foreign Key for each entity. Normally, the Primary Key for each entity will be underlined, and any Foreign Keys will have a star next to the name. Below is an example of an ER Diagram for Customers and Cars.

Additional, you should name the relationship between entities, in this case the relationship has been named 'owns' because one customer 'owns' many cars.
