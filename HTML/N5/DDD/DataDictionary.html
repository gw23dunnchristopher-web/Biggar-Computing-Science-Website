Data Dictionaries

What is a Data Dictionary?

Data Dictionaries are created during the design phase to define the structure of a database. They contain all of the elements to be present in the database once it is implemented.

The data dictionary should include:

  • The names of any entities (tables) in the database.
  • The names of the attributes (fields) of each entity.
  • Primary and Foreign Keys.
  • The data type of each attribute.
  • The size of any of the text attributes.
  • Whether or not any of the attributes are required.

Here is an example of a data dictionary for a database with two entities - Cars and Customers:

Data Dictionary Breakdown

Entity Name

This will be the name of one of the tables that is being used in the database. It is important to give the table a name that describes what it contains. In the case of the example above, one of the tables is called 'Cars' so we know that it contains information about cars, and the other is called 'Customers', so that we know that it contains information about customers.

Attribute Names and Types

Attributes are the names of the fields in each table. These fields should be given appropriate names as well to identify the data that they will contain. For example, the Cars entity has the the following attributes: License No., Make, Model, Colour, Price and Customer ID. Each attribute must be assigned a type of data that can be entered into it, this will be stated next to the name of the attribute in the data dictionary. The attribute types are listed in the table below:

Attribute Types
Type Description Example
Text Letters, numbers and symbols Yellow, SC11 THZ, bob@emailacct.com
Number Numbers only 1, 0.05, -912
Date Day, Month, Year 11/12/24, 11th December 2024
Time Hours, Minutes, Seconds 13:00, 13:00:00, 1.00pm
Boolean Stores a Boolean value, 1 or 0, yes or no, true or false. It can be represented by a checkbox. Yes, No, True, False

Key

This is used to identify whether or not the attribute is a Primary Key or a Foreign Key. The Primary Key is the attribute that uniquely identifies each record in the table, and the Foreign Key is the Primary Key from another table used to link the two tables together.

In a data dictionary, a Primary Key is represented by PK and a Foreign Key is represented by FK.

Size

Text fields can normally contain up to 255 characters, letters, or symbols. Using the full amount of characters for every text field will make your database take up more space than it needs to. In order to reduce the amount of space used, you can limit the number of characters that can be entered into each field by putting a limit on it. This number is what you will enter into the Size column of your data dictionary.

In the example dictionary above, you can see that the 'First Name' attribute has a 30 in this column. That means that the user cannot enter more than 30 characters in that field (it is unlikely that someone's first name would be more than 30 characters long.)

Required

This is used to identify whether or not the attribute is required to be filled in. If it is required, then the user will not be able to add a new record to the database until they have entered a value into that field. In the data dictionary, any field which must be entered before the record can be added will have a Y or Yes in this column.

Validation

This is used to identify any validation that needs to be done on the data entered into the field. There are four types of validation that you will need to be aware of:

Validation Types
Type Description Example
Range Check This is used to specify a range of numbers that can be entered into the field. For example, if you had a field for a person's age, you would only want to allow numbers between 0 and 120 to be entered.
Length Check This will require the user to enter a certain number of letters before they can add the record. For example, if you were entering a postcode, you would need to enter at least 5 characters for it to be valid.
Restricted Choice This will require the user to choose from a list of options. For example, if you were entering a person's gender, you would only want them to be able to choose from Male, Female or Other. The options that you will be able to choose from should be listed in the data dictionary.
Presence Check This has already been covered in the required column, as it determines whether or not the field must be filled in before the record can be submitted. If a presence check is required, just put "Yes" in the Required column of the data dictionary.
