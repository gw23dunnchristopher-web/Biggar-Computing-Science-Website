Creating a Data Dictionary

A Worked Example

Consider the following example:

A local furniture company holds info in its products and its designers. It stocks tables, chairs and desks. They hold the item number, name and description, it’s price and type and also the amount of it they have in stock. They will also store the information on the designer including their ID, name, email address and phone number.

From this, we can get the following attributes:

  • Item Number
  • Item Name
  • Item Description
  • Item Price
  • Item Type
  • Amount of Item in Stock
  • Designer ID
  • Designer Name
  • Designer Email
  • Designer Phone

Step 1 - Identify the Entities

From the list of attributes we have create, we can identify to distinct entities - items and designers. Each one of the attributes would fall into one of those two categories.

Step 2 - Identifying the Attributes

Now we have to decide which of the attributes in our list below to each entity. Attributes concerning the items will belong to the Items entity, and the rest will belong to the Designers entity.

Step 3 - Identify the Relationship

There will be a relationship between the two entties, and we have to decide what it is.

  • One item can only have one designer.
  • However, a designer can design many items.
  • So it will be a one to many relationship between the Item and Designer.

In the image below, you can see that I have added a prong to the side of the relationship line on the side of the relationship entity. This shows that there can be one designer for many items. I have named this relationship 'design' because one designer can design many items.

Step 4 - Identify the Field Types and Sizes for Entity 1

Now we need to decide what type of data each attribute will contain, and the size of the field if it is a text field. We will start with the Designer entity because each item will only have one designer.

  • Designer ID - Text (Sample Data: MSMI01), this will have a maximum of 6 characters.
  • Designer Name - Text (Sample Data: Mary Smith), this will have a maximum of 20 characters - it is unlikely our database will have longer names.
  • Designer Email - Text (Sample Data: mary.smith@email.com) this will have a maximum of 30 characters - it is unlikely our database will have longer emails than that.
  • Designer Phone - Text (Sample Data: 07700900989), this will also be text because using number would not include the first 0 in the phone number, phone numbers are 11 numbers long.

We will use this information to start filling in the data dictionary:

Step 5 - Identify any Keys in Entity 1

A Primary Key is a field that is unique to each record in the entity. In this case it would be unique to each Designer - the Designer ID is unique to each designer, so we will add PK in the Key column to show that this is the Primary Key. A Foreign Key is a Primary Key in another entity that links the two entities together - since there is only one designer for each item, there will be no Foreign Key in the Designers table.

Step 6 - Add any Validation Checks for Entity 1

Now we need to add any validation checks that need to be done on the data entered into the fields:

  • Primary keys are ALWAYS required, so we will put a Yes in the Required Column.
  • In order to record information about a designer, we will at the very least need their name and phone number, so those will also be required.
  • A valid phone number will always be 11 numbers long, so we will need to include a length check to make sure that the user enters 11 characters.

The data dictionary below has been updated with the Validation Checks for the Designers entity.

Step 7 - Identify the Field Types and Sizes for Entity 2

We will now do the same for the Items entity, getting the field types and sizes (where appropriate).

  • Item Number - Number (Sample Data: 1).
  • Item Name - Text (Sample Data: Oak Table), this will have a maximum of 20 characters.
  • Description - Text (Sample Data: A large oak table), this will have a maximum of 50 characters.
  • Price - Number (Sample Data: £150).
  • Type - Text (Sample Data: Table), this will have a maximum of 10 characters.
  • Amount - Number (Sample Data : 6).

As each of these items will have a designer, we should also include the Designer ID field from the Designers Entity. It will have the same size and type as it does in that entity (Text, size 6).

This information can be placed at the bottom of the data dictionary.

Step 8 - Identify any Keys in Entity 2

The unique identifier for each item is Item No, so this will be the Primary Key. There is a Foreign Key this time too, as the Primary Key from the Designers Entity is also in the Items entity to link the two tables together. In addition to placing a PK next to Item No, we should also place an FK next to Designer ID in the Items Entity to show that it is a Foreign Key.

Step 9 - Add any Validation Checks for Entity 2

We now need to add the validation checks for the Items entity.

  • As the Primary Key, the Item Number will be required.
  • The Item will also need a name for us to record it, so the Item Name will also be required.
  • The price for an item would have to be greater than 0, since there can't be any item sold for a negative price.
  • The type of the item should be restricted to the types of items that exist, in this case "table, chair and desk".
  • There also cannot be a negative number of items, but there can be 0 left in stock. So this field must be greater than or equal to 0.
  • The list of designer IDs will be looked up from the Designer table. This field will also be required since each item must have a designer.
